2 ЗАВДАННЯ

     Потрібно провести дослідження та аналіз офіційних стандартів написання програмного коду для вибраної мови програмування, а також продемонструвати їх практичне використання на конкретних прикладах.
     Студент має ознайомитися з офіційними coding conventions та style guides обраної мови, визначити основні принципи оформлення коду: правила найменування пакетів, класів, методів, змінних і констант; вимоги до форматування (відступи, розміщення дужок, пробіли); особливості структурування класів і модулів; ідіоматичні підходи мови; рекомендації щодо створення читабельного та підтримуваного коду.
     Для кожного правила оформлення необхідно підготувати приклади, що демонструють як коректне, так і некоректне його застосування, з поясненням, чому дотримання цього стандарту є важливим. На основі виконаного аналізу студент повинен створити презентацію українською мовою, що містить щонайменше 10 слайдів. У вступі слід надати коротку характеристику обраної мови програмування, описати її появу, розвиток і сфери використання. Основна частина має включати докладний розгляд стандартів оформлення коду з прикладами.У висновках необхідно оцінити значення дотримання стилістичних вимог та їхній вплив на якість програмного забезпечення. 
     Готову презентацію потрібно завантажити на платформу dl.nure.ua, а приклади коду - розмістити у репозиторії GitHub із детальною документацією.
      

3 ОПИС ВИКОНАНОЇ РОБОТИ
     3.1 Вибір мови програмування та джерел
     Для виконання практичної роботи обрано мову програмування Go (Golang) — сучасну компільовану мову, розроблену інженерами компанії Google (Robert Griesemer, Rob Pike та Ken Thompson) у 2007 році та офіційно представлено у 2009 році. Go було створено з метою забезпечення високої продуктивності, простоти синтаксису та ефективної роботи з багатопоточністю. Мова швидко стала популярною у сфері серверної розробки, мікросервісів, системного програмування та хмарних технологій. Її стандарти оформлення коду визначаються офіційними інструментами (gofmt, go vet) та рекомендаціями від команди розробників, що забезпечує єдину стилістику написання програм.
     3.2 Іменування пакунків: Чіткість та простота
     Назви пакунків (packages) повинні бути короткими, у нижньому регістрі та однині. Уникайте використання символу підкреслення (_). Чим простіша назва пакунка, тим чистіший код, що його використовує, оскільки назва пакунка входить у частину виклику.
     
1  // ПОГАНО
2  // Назва з символом підкреслення і не в однині
3  package customer_manager 
4
5  // Приклад виклику:
6  // customer_manager.New( )


1  // ДОБРЕ
2  // Коротка назва в нижньому регістрі та однині
3  package user 
4
5  // Приклад виклику:
6  // user.New( )
     3.3 Іменування: Експортовані та неекспортовані елементи
     У Go видимість елементів визначається регістром першої літери назви. Це є ключовою конвенцією, що дозволяє чітко дотримуватись правил доступу та обсягу видимості.
     Експортовані елементи (Public): Використовуйте PascalCase (з великої літери). Для елементів, які повинні бути доступні за межами пакунка (публічні).
     Неекспортовані елементи (Internal/Private): Використовуйте camelCase (з малої літери). Для елементів, призначених лише для внутрішнього використання пакунка (приватні).
     Дотримання цієї конвенції запобігає випадковому експортуванню внутрішніх структур та функцій, підтримуючи чистоту API вашого пакунка.

1  // Експортовані елементи (Public) - PascalCase
2  type CustomerData struct {
3      // Поля також експортуються, якщо починаються з великої літери
4      Name string
5      Email string
6  }
7  
8  // Виклик з іншого пакунка:
9  // data := mypackage.CustomerData{Name: "John"}

1  // Неекспортовані елементи (Internal/Private) - camelCase
2  type customerData struct {
3      // Поля не експортуються, якщо починаються з малої літери
4      name string
5      email string
6  }
7  
8  // Доступно лише всередині поточного пакунка
9  // data := customerData{name: "John"}

     3.4 Іменування: Скорочення та контекст змінних
     Go надає перевагу стислості та чіткості. Довгі назви змінних (наприклад, maximumConnections) часто зайві, якщо контекст використання зрозумілий. Важливо знайти баланс між короткістю та змістовністю.
     Ключові принципи:
- Змістовні Скорочення: Використовуйте короткі, але змістовні назви змінних, такі як maxConns замість maximumConnections.
- Однолітерні Імена: Для лічильників та простих ітераторів (наприклад, у циклах) підходять однолітерні назви (i, j, k).
- "Качиний" Префікс (Duck Typing Prefix): Уникайте використання "качиного" префіксу (наприклад, strName для рядка або intAge для цілого числа).

1  // ПОГАНО
2  // Надмірно довгі та надлишкові назви
3  const maximumConnections = 100 
4  
5  // Надмірно довга назва ітератора
6  for index_of_loop := 0; index_of_loop < 10; index_of_loop++ {
7      // ...
8  }

1  // ДОБРЕ
2  // Короткі, але зрозумілі скорочення
3  const maxConns = 100 
4
5  // Короткі однолітерні імена для простих ітераторів
6  for i := 0; i < 10; i++ { 
7      // ...
8  }
     3.5 Коментарі: Документування експортованих елементів
     Коментарі у Go — це не просто пояснення, а невіддільна частина публічного API вашого коду, яка використовується інструментом godoc.
     Обов'язковість: кожен експортований (публічний) елемент (функція, змінна, константа, тип) повинен мати коментар документації.
     Формат: коментар має починатися з назви елемента, який він документує. Це дозволяє godoc коректно генерувати документацію та легко ідентифікувати елемент.
     
1  // ПОГАНО
2  // Створює новий сервер
3  func NewServer() *Server { ... }
4
5  // Коментар не починається з назви NewServer, що порушує конвенцію godoc

1  // ДОБРЕ
2  // NewServer створює та ініціалізує новий HTTP-сервер,
3  // готовий до обробки запитів.
4  func NewServer() *Server { ... }
5
6  // Коментар чітко описує функцію та починається з її назви

     3.6 Коментарі: Пояснення "Чому" та поглиблення логіки
     Ефективні коментарі не повторюють код, а пояснюють мотивацію та складні аспекти реалізації, відповідаючи на питання "Чому?".
     Пояснення причин: використовуйте коментарі, щоб пояснити причини вибору певної логіки, алгоритму або обхідного шляху (workaround).
     Складні ділянки: очевидний код не потребує коментарів. Коментуйте лише складні або неочевидні ділянки, які можуть заплутати майбутніх розробників.
     Такі коментарі є цінним ресурсом для розуміння архітектурних рішень та технічного боргу.

1  // ПОГАНО
2  // Перевіряємо, чи користувач адміністратор
3  if user.Role == "Admin" { ... }
4  
5  // Коментар просто повторює те, що очевидно з коду

1  // ДОБРЕ
2  // Використовуємо тут бінарний пошук, оскільки 
3  // масив 'data' завжди відсортований, що забезпечує складність 
4  // O(log n).
5  result, err := searchSorted(data, target)
6  
7  // Коментар пояснює неочевидну причину вибору алгоритму

     3.7 Форматування: gofmt та goimports для чистоти коду
     Автоматичне форматування за допомогою gofmt та goimports є фундаментальною практикою в Go. Це усуває суперечності щодо стилю та забезпечує єдність коду в команді.
     gofmt: * Застосовує офіційні правила форматування Go.
- Стандартизує відступи (використовуючи табуляцію), пробіли та загальну структуру коду.
- Робить код послідовним і легким для читання.
     
     goimports:
- Це додаток до функціоналу gofmt.
- Автоматично сортує імпорти, додає необхідні та видаляє невикористовувані пакети.
- Підтримує чистоту файлу імпортів.
     
Приклад: Добре відформатований код
     
1  func Calculate(a, b int) {
2  	result := a + b
3  	return result
4  }
     
     3.8 Конвенції щодо інтерфейсів: Зменшення залежностей
     Правило: Інтерфейси мають бути маленькими (принцип "Single Responsibility Principle") і зазвичай визначатися в пакунку, який їх використовує (принцип "Приймай інтерфейс").
     Ключові принципи:
     Маленькі інтерфейси:
- Великі інтерфейси ("God interfaces") ускладнюють тестування та призводять до жорстких залежностей.
- Go заохочує інтерфейси з 1-2 методами.
     Принцип "Приймай інтерфейс":
- Інтерфейс повинен визначатися не там, де він реалізується, а там, де він використовується.
- Це зменшує залежність між пакунками.
     Гнучкість:
- Типи неявно задовольняють невеликі інтерфейси, що робить дизайн більш гнучким.

1  // ПОГАНО
2  // interface.go (визначений у пакунку 'storage')
3  type DatabaseInterface {
4      Save() error      // Потрібний
5      Load() error      // Непотрібний для поточного використання
6      Connect() error   // Непотрібний для поточного використання
7  }
8  
9  func SaveUser(db DatabaseInterface) error {
10     return db.Save()
11 }
12 // Функція SaveUser залежить від непотрібних методів Load та Connect

1  // ДОБРЕ
2  // user_service.go (визначений у пакунку 'service')
3  type DataSaver interface {
4      Save() error // Лише необхідний метод
5  }
6
7  func SaveUser(saver DataSaver) error {
8      return saver.Save() // Залежність лише від Save()
9  }
10 // Функція SaveUser приймає лише DataSaver, що зменшує залежність

     3.9 Функції: Простота та обробка помилок
     Функції в Go мають бути мінімалістичними та цілеспрямованими. Це не лише полегшує читання, але й спрощує тестування та відлагодження.
     Ключові принципи:
     Єдина відповідальність:
- Кожна функція повинна виконувати одну, чітко визначену задачу.
- Розбивайте складні операції на менші, керовані функції.
     Обробка помилок:
- Помилки завжди мають бути останнім значенням, що повертається.
- Ідіома if err != nil є стандартом для обробки помилок у Go.
     Легкість тестування:
- Короткі функції з чіткими входами та виходами набагато легше тестувати, що сприяє створенню надійного коду.

     Приклад: коректна обробка помилок
     
1  func ReadData(path string) ([]byte, error) {
2  	data, err := os.ReadFile(path)
3  	if err != nil {
4  		// Повертаємо помилку із контекстом, використовуючи fmt.Errorf
5  		return nil, fmt.Errorf("Read: %w", err) 
6  	}
7  	return data, nil
8  }
     


ВИСНОВКИ

     Виконання практичної роботи дозволило ґрунтовно дослідити офіційні стандарти оформлення коду в Go (Golang). Було проаналізовано ключові конвенції, що сприяють підвищенню якості коду: чітке іменування пакунків (коротко, в однині та нижньому регістрі) та використання регістру для визначення доступу (великі літери для експорту, малі — для внутрішнього використання). Ми також підтвердили важливість принципу стислості у назвах змінних, обов'язковість коментування публічних елементів для інструменту godoc, а також необхідність пояснювати мотивацію складних рішень, а не лише їхню суть. Крім того, було вивчено принцип єдиної відповідальності для функцій, стандарти обробки помилок (if err != nil) та доцільність використання малих інтерфейсів для досягнення модульності.
     На основі отриманих знань можна зробити висновок, що систематичне застосування цих офіційних правил Go, підкріплене автоматичними інструментами форматування (gofmt, goimports), є критичним елементом професійної розробки. Дотримання єдиного стилю значно підвищує читабельність, надійність та підтримуваність коду, мінімізує технічний борг та спрощує спільну роботу в команді.
